# 开发总结：流式识别（失败，暂搁置）

## 开发项背景

原有语音输入是"录完再识别"模式，用户需要松开热键后才能看到识别结果。希望实现"边说边出字"的流式识别体验。

## 尝试过程

### 尝试一：伪流式（SenseVoice 定期重新推理 + BackSpace 替换）

**方案**：录音过程中每隔 ~1 秒，对已积累的全部音频运行一次 SenseVoice 完整推理，用 BackSpace 删除上次的中间结果，再粘贴新结果。

**失败原因**：
1. BackSpace 替换机制本质脆弱——依赖光标位置不变，如果用户在录音过程中做了任何操作（甚至只是焦点切换），文本就会错乱
2. 跨会话状态泄漏——第二次按键识别的结果 B 会把第一次的结果 A 删掉，因为 `prev_length` 没有正确隔离
3. `xdotool key BackSpace BackSpace ...` 参数传递方式有 bug（拼成单字符串而非多参数）

**结论**：BackSpace 替换不是可靠的文本更新机制，这个方向不可行。

### 尝试二：Paraformer-streaming 真流式

**方案**：使用 FunASR 的 Paraformer-streaming 模型，每个 600ms 音频 chunk 增量输出新文字，直接追加到焦点窗口，不需要 BackSpace。

**失败原因**：
1. **字符大量重复**——我们自己做了 buffer 分 chunk，但模型内部也通过 `cache["prev_samples"]` 管理 buffer，导致音频被双重处理，同样的文字输出两遍
2. **去掉自有 buffer 后无流式输出**——直接把小块音频交给模型后，中间不再有任何输出，所有文字在 `end_stream(is_final=True)` 时才一次性出现，失去了流式的意义
3. **无标点符号**——Paraformer-streaming 是纯 ASR 模型，不带标点。需要额外的标点恢复模型：
   - `ct-punc`（1.2GB）——太大，接近一个完整 ASR 模型的大小
   - `ct-punc-c`（300MB）——较小，但效果未验证，且标点模型可能干扰流式输出的增量逻辑
4. **识别质量差**——实测中文识别有大量重复字，整体效果远不如 SenseVoice 离线模式

## 调研发现

在开发过程中调研了多个方案：

### 模型对比

| 模型 | 流式 | 自带标点 | 中英混合 | 备注 |
|------|------|---------|---------|------|
| SenseVoice-Small | ❌ | ✅ | ✅ | 当前在用，效果好 |
| Paraformer-streaming | ✅ | ❌ | 弱 | 需额外标点模型，实测效果差 |
| Qwen3-ASR (0.6B/1.7B) | ✅（需 vLLM） | ✅ | ✅ | 准确率最高，但需引入 vLLM 重依赖 |
| Fun-ASR-Nano (0.8B) | ❌（暂无本地流式 API） | ✅ | ✅ | 最新模型，流式仅云端 API |

### Qwen3-ASR 的优势

- AISHELL-2 CER：Qwen3-ASR-1.7B (2.71) > Paraformer-Large (2.85) > SenseVoice-L (3.04)
- 同一模型同时支持流式和离线推理
- 0.6B 版本仅需 ~1.2GB 显存
- 流式 vs 离线精度损失约 18%，仍然是工业级水平
- 支持 30 种语言 + 22 种中文方言

### Qwen3-ASR 的限制

- 流式推理必须使用 vLLM 后端（重量级依赖）
- 使用自己的 `qwen-asr` 框架，不兼容 FunASR 的 `AutoModel`
- 与当前 SenseVoice/FunASR 生态不同，引入需要较大改动

## 当前状态

代码已全部还原到初始实现状态，流式识别功能未合入。工具仍以 SenseVoice 离线模式正常工作。

## 后续建议

流式识别应等到引入 Qwen3-ASR 时一并解决：
1. Qwen3-ASR 是目前唯一同时满足"流式 + 自带标点 + 高准确率 + 中英混合"的开源模型
2. 需要引入 vLLM 作为推理后端
3. 可同时保留 SenseVoice 作为轻量级离线选项（小显存设备）
4. 也可关注 Fun-ASR-Nano 的本地流式 API 是否会开放
